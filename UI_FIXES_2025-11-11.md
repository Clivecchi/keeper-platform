# UI Fixes - Authentication & Board Edit Mode
**Date:** November 11, 2025  
**Issues Addressed:**
1. Dashboard link not working from public board view
2. Board appearing the same for logged-in users as public users
3. Authentication persistence requiring re-login when navigating to /root

## Root Cause Analysis

### Issue 1: Authentication Not Persisting in AuthContext

**Problem:** The `AuthContext` in production mode was not fetching user data from the server. It relied on the `AuthGate` component to validate sessions, but the two components didn't share state properly.

**Location:** `apps/web/src/context/AuthContext.tsx`

**What was happening:**
- `AuthGate` would validate the session with `/api/kam/auth/me` and store user data locally
- `AuthContext` (which manages global auth state) was NOT fetching user data in production mode
- Result: Components using `useAuth()` would see `isAuthenticated: false` even when cookies were valid

**Fix Applied:**
Modified `AuthContext` to fetch user data from the server in production mode on mount:

```typescript
// Old behavior (BROKEN):
if (IS_PROD) {
  console.log('[AuthContext] Production mode: skipping localStorage (using cookies)');
  setIsLoading(false);
  return; // âťŚ Never fetched user data!
}

// New behavior (FIXED):
if (IS_PROD) {
  (async () => {
    try {
      const apiUrl = (import.meta as any)?.env?.VITE_API_URL || 'https://api.ke3p.com';
      const response = await fetch(`${apiUrl}/api/kam/auth/me`, {
        method: 'GET',
        credentials: 'include', // âś… Send HttpOnly cookie
      });

      if (response.ok) {
        const data = await response.json();
        if (data.user) {
          setUser(data.user);
          setToken('cookie-based'); // âś… Indicate authenticated
        }
      }
    } finally {
      setIsLoading(false);
    }
  })();
}
```

### Issue 2: Board Edit Controls Not Visible

**Problem:** The board view logic was correct, but because authentication wasn't working (Issue 1), users appeared as "guest" even when logged in.

**Location:** `apps/web/src/pages/d/PublicDomainPage.tsx`

**What was happening:**
- The edit button only shows when `isAuthenticated && isDomainAdmin && !isEditMode`
- Because `isAuthenticated` was false (due to Issue 1), edit controls never appeared
- Admin frames were filtered out for non-authenticated users

**Fix Applied:**
The logic was already correct. With Issue 1 fixed, authentication now works and:
- âś… `isAuthenticated` will be `true` for logged-in users
- âś… `isDomainAdmin` is fetched from API based on user's role
- âś… Edit button appears when appropriate
- âś… Admin frames are visible to domain admins

Added debug logging to help troubleshoot:
```typescript
console.log('[PublicDomainPage] User permissions:', { 
  user: user.email, 
  role, 
  canEdit, 
  isDomainAdmin: isAdmin,
  totalFrames: boardDataResponse.board?.frames?.length || 0
});
```

### Issue 3: Dashboard Link Navigation

**Problem:** The Dashboard link was working correctly, but because authentication wasn't persisting, navigating to `/root` would trigger the protected route guard and redirect to `/login`.

**Location:** `apps/web/src/pages/d/PublicDomainPage.tsx` (line 96-98)

**What was happening:**
- Dashboard button calls `navigate('/root')`
- `/root` is a protected route (requires authentication)
- Because `isAuthenticated` was false, the `ProtectedRoute` component redirected to `/login`

**Fix Applied:**
The navigation logic was already correct. With Issue 1 fixed:
- âś… Authentication persists across navigation
- âś… `ProtectedRoute` sees valid auth state
- âś… Dashboard loads without requiring re-login
