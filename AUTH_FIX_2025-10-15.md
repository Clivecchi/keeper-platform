# Authentication Fix - October 15, 2025

## Executive Summary

**Issue #1**: Authenticated endpoints (`/api/whoami`, `/api/kip/user-keys`) returned `401 Unauthorized` despite users being logged in.

**Root Cause #1**: Cookie name mismatch between login flow and authentication middleware.

**Fix #1**: Added `keeper_session` to the cookie candidates list in `attachUser` middleware.

**Issue #2**: After fixing cookie issue, login returned "An unknown error occurred" in the frontend.

**Root Cause #2**: Response format mismatch - backend returned `{ ok: true, token, user }` but frontend expected `{ success: true, data: { token, user } }`.

**Fix #2**: Updated login endpoint to return standardized response format matching frontend expectation.

**Status**: ✅ **RESOLVED** - Two-part fix deployed.

---

## Problem Statement

Users reported that authenticated API endpoints were returning `{ "error": "Unauthorized" }` even after successful login:

1. `GET /api/whoami` → `401 Unauthorized`
2. `GET /api/kip/user-keys` → `401 Unauthorized`

Meanwhile:
- Login flow worked correctly
- `/api/kam/auth/me` endpoint worked correctly
- Public endpoints worked correctly
- Model provider test worked correctly

---

## Problem Statement - Issue #2 (Login Error)

After deploying Fix #1 (cookie name fix), users reported:
- Login form shows "An unknown error occurred"
- Console shows `✅ Login success: Response {status: 200, ok: true, …}` (multiple times)
- A 404 error appears: `login:1 Failed to load resource: the server responded with a status of 404`

The login API call succeeds (200 OK) but the frontend thinks it failed, causing retry loops.

## Root Cause Analysis

### The Cookie Name Mismatch

The authentication system had an **inconsistency in cookie naming**:

#### Login Flow (Working)
**File**: `apps/api/src/kam/auth.ts` + `apps/api/src/kam/session.ts`

```typescript
const COOKIE_NAME = 'keeper_session';  // Line 8 in session.ts

export async function login(req: Request, res: Response) {
  // ... authenticate user ...
  const token = jwt.sign({ userId: user.id, email: user.email }, JWT_SECRET, { expiresIn: '7d' });
  
  // Sets cookie named 'keeper_session'
  setSessionCookie(req, res, token);
  // ...
}
```

✅ Login sets cookie: **`keeper_session`**

#### Global Auth Middleware (Broken)
**File**: `apps/api/src/middleware/auth.ts`

```typescript
// BEFORE FIX:
const COOKIE_CANDIDATES = ['token', 'keeper_token', 'auth_token'];

function getToken(req: Request): string | null {
  const auth = req.headers.authorization || '';
  if (auth.startsWith('Bearer ')) return auth.slice(7);
  const cookies = (req as any).cookies || {};
  for (const name of COOKIE_CANDIDATES) if (cookies[name]) return cookies[name];
  return null;
}

export function attachUser(req: Request, _res: Response, next: NextFunction) {
  // ... 
  const token = getToken(req);  // Never finds 'keeper_session' cookie!
  // ...
}
```

❌ Middleware looks for: **`token`**, **`keeper_token`**, **`auth_token`** — but NOT **`keeper_session`**!

### Why Some Routes Worked

The codebase has **two different auth middleware implementations**:

1. **`attachUser`** (in `middleware/auth.ts`)
   - Applied globally to all requests (line 298 in `index.ts`)
   - Used by: `/api/whoami`, `/api/kip/user-keys`, most routes
   - **Did NOT check for `keeper_session` cookie** ❌

2. **`authWeb`** (in `kam/session.ts`)
   - Applied to specific KAM routes (line 16 in `kam/auth-routes.ts`)
   - Used by: `/api/kam/auth/me`
   - **Correctly reads `keeper_session` cookie** ✅

This explains why `/api/kam/auth/me` worked but `/api/whoami` didn't!

### The Response Format Mismatch (Issue #2)

After fixing the cookie issue, a second problem emerged:

#### Backend Response (KAM Auth)
**File**: `apps/api/src/kam/auth.ts` (line 48-57, BEFORE fix)

```typescript
return res.json({
  ok: true,      // ❌ Backend returns "ok"
  token,
  user: { ... }
});
```

#### Frontend Expectation
**File**: `apps/web/src/components/AuthForm.tsx` (line 42-51)

```typescript
const result = await apiFetch('/api/kam/auth/login', { ... });

if (result.success) {  // ❌ Frontend checks "success"
  auth.login(result.data);
  navigate('/root');
} else {
  setError(result.error?.message || 'An unknown error occurred.');
}
```

**The Problem**:
- Backend returns `{ ok: true, token, user }` (flat structure)
- Frontend expects `{ success: true, data: { token, user } }` (nested structure)
- Frontend check `if (result.success)` fails because `result.success` is undefined
- Error message: "An unknown error occurred"

**Why this mismatch existed**:

There are **two login endpoints** in the codebase:

1. **`/api/kam/auth/login`** (in `kam/auth.ts`)
   - Returns: `{ ok: true, token, user }` ❌

2. **Inline login** (in `index.ts` line 627-678)
   - Returns: `{ success: true, data: { user, token } }` ✅

The frontend was updated to use the KAM auth endpoint but the response format wasn't aligned.

---

## The Fix

### Fix #1: Cookie Name Alignment

**File**: `apps/api/src/middleware/auth.ts`

```typescript
// BEFORE:
const COOKIE_CANDIDATES = ['token', 'keeper_token', 'auth_token'];

// AFTER:
const COOKIE_CANDIDATES = ['keeper_session', 'token', 'keeper_token', 'auth_token'];
```

**Impact**: The `attachUser` middleware now checks for the `keeper_session` cookie first, enabling it to find the authentication token set by the login flow.

### Fix #2: Response Format Alignment

**File**: `apps/api/src/kam/auth.ts`

```typescript
// BEFORE:
return res.json({
  ok: true,
  token,
  user: { id, email, name, avatar_url },
});

// AFTER:
return res.json({
  success: true,
  data: {
    token,
    user: { id, email, name, avatar_url },
  },
});
```

**Impact**: Login endpoint now returns the format expected by the frontend, eliminating the "unknown error" message and allowing successful login flow completion.

### Why These Fixes Work

**Fix #1 (Cookie)**:
1. Login flow sets `keeper_session` cookie (unchanged)
2. Global middleware now reads it: `attachUser` checks `keeper_session` first
3. Backward compatible: Still checks legacy cookie names as fallback
4. Minimal change: Single-line fix, no breaking changes
5. Consistent with session module: Aligns with `kam/session.ts` implementation

**Fix #2 (Response Format)**:
1. Login endpoint returns standardized format: `{ success: true, data: { token, user } }`
2. Frontend can now properly detect successful login: `if (result.success)` evaluates to true
3. Auth context receives correct data structure: `auth.login(result.data)` gets `{ token, user }`
4. Navigation proceeds: `navigate('/root')` executes after successful login
5. Consistent with other endpoints: Matches format used by inline `/api/kam/auth/login` in `index.ts`

## Technical Details

### Authentication Flow (After Fix)

```
┌─────────────────────────────────────────────────────────────────┐
│                        User Login Flow                          │
└─────────────────────────────────────────────────────────────────┘
  1. POST /api/kam/auth/login { email, password }
  2. Server validates credentials
  3. Server generates JWT: { userId, email }
  4. Server sets cookie: keeper_session=<JWT>
     ├─ HttpOnly: true
     ├─ Secure: true
     ├─ SameSite: lax
     ├─ Domain: .ke3p.com
     └─ MaxAge: 7 days

┌─────────────────────────────────────────────────────────────────┐
│                   Subsequent API Requests                        │
└─────────────────────────────────────────────────────────────────┘
  1. Browser sends request with Cookie: keeper_session=<JWT>
  2. Express cookieParser() parses cookies
  3. attachUser middleware runs (global, line 298 index.ts):
     a. Checks req.session.user (not set)
     b. Calls getToken(req):
        i.   Checks Authorization header (not present)
        ii.  Checks cookies for 'keeper_session' ✅ FOUND
        iii. Returns JWT token
     c. Verifies JWT and extracts { id, email }
     d. Sets req.user = { id, email, roles }
  4. Route handler receives req.user
  5. Returns authenticated response

┌─────────────────────────────────────────────────────────────────┐
│                      Cookie Priority Order                       │
└─────────────────────────────────────────────────────────────────┘
  1. keeper_session  ← Primary (used by current login)
  2. token           ← Legacy fallback
  3. keeper_token    ← Legacy fallback
  4. auth_token      ← Legacy fallback
```

### JWT Payload Structure

```json
{
  "userId": "491307f3-b331-436c-b53a-09a11ec110cb",
  "email": "user@example.com",
  "iat": 1697328000,
  "exp": 1697932800
}
```

**Note**: The middleware handles both `id` and `userId` fields for backward compatibility:
```typescript
const id = (payload as any)?.id || (payload as any)?.userId;
```

## Prevention & Best Practices

### Lessons Learned

1. **Centralize cookie names**: Define cookie names in a shared constant file
2. **Test auth across all middleware**: Ensure consistency between middleware implementations
3. **Document auth flow**: Clearly document which middleware is used by which routes
4. **Use TypeScript constants**: Prevent typos with compile-time checks

### Recommended Follow-Up

#### 1. Consolidate Auth Middleware (Future)

Currently there are **multiple auth middleware implementations**:
- `attachUser` in `middleware/auth.ts`
- `authWeb` in `kam/session.ts`
- `authMiddlewareCompat` in `middleware/authMiddleware.ts`
- `kamOrUserAuth` in `middleware/auth-combined.ts`

**Recommendation**: Create a single, canonical auth middleware that all routes use.

#### 2. Create Shared Auth Constants

**File**: `apps/api/src/shared/auth-constants.ts` (suggested)

```typescript
export const AUTH_CONFIG = {
  COOKIE_NAME: 'keeper_session',
  COOKIE_DOMAIN: process.env.COOKIE_DOMAIN || '.ke3p.com',
  JWT_SECRET: process.env.JWT_SECRET!,
  TOKEN_EXPIRY: '7d',
  
  // Legacy cookie names for backward compatibility
  LEGACY_COOKIE_NAMES: ['token', 'keeper_token', 'auth_token'],
} as const;
```

#### 3. Add Integration Tests


## Conclusion

This issue required a **two-part fix** to resolve critical authentication problems:

### Fix #1: Cookie Name Alignment
A single-line change that aligned cookie naming between the login flow (`keeper_session`) and the global auth middleware. Simply added the current cookie name to the list of candidates the middleware checks.

### Fix #2: Response Format Standardization
Updated the KAM login endpoint to return the standardized response format that the frontend expects, eliminating the "unknown error" message.

**Both fixes are minimal, surgical, and backward-compatible**:
- Fix #1: One line added to cookie candidates array
- Fix #2: Response structure change (no breaking changes to downstream consumers)
- No database migrations required
- No frontend changes required
- Maintains CLI/tool compatibility
